# task_1 수행계획서

## 작업 제목

원격 게임 서버 오픈 / 점프 접속 기능 계획 구체화

## 목표

개인 서버가 공유기 뒤 내부망에 있어도 사용자가 설치형 프로그램 UI에서 클릭만으로 게임 서버를 외부에 공개할 수 있도록, 현재 기획 문서를 기준으로 구현 전 분석과 설계 범위를 정리한다.

## 현재 현상 또는 요구사항

현재 [원격 게임 서버 오픈 / 점프 접속](../remote-game-server-access.md) 문서에는 다음 네트워크 유형과 추천 구현 방식이 정리되어 있다.

- 직접 외부 오픈
- 중앙 Relay/Jump 서버를 통한 점프 서버 방식
- 외부망 서버에서 내부망 서버로 전달하는 내부망 점프 방식
- 1순위 구현 방식: Agent + FRP
- 2순위 구현 방식: Nginx Stream

작업지시자는 이 문서를 계획서로 보고 있으며, `AGENT.md` 절차에 맞춰 프로젝트 작업을 진행하기를 요청했다.

추가 피드백으로 다음 기준이 확정되었다.

- 사용자 접점은 웹 UI가 아니라 설치형 프로그램 UI다.
- 현재 작업 공간에서는 수동 TCP 검증이 불가능하므로, 추후 Git 레포가 추가될 때 검증 절차를 넣을 수 있도록 준비한다.
- UDP와 Nginx Stream은 실제 접속 후 원격 수정이 어려운 영역이므로, 우선 사용자 안내 가이드 제공 대상으로 본다.

문서 기반 작업이므로 다음 파일을 확인한다.

- `AGENT.md`
- `docs/remote-game-server-access.md`
- `docs/orders/20260428.md`

향후 구현 단계에서는 실제 코드베이스와 Git 레포가 추가되거나 연결된 뒤, Agent, Relay, Docker 실행 경로를 기준으로 재현 가능한 검증 절차를 작성한다.

## 원인 가설

현재는 구현 문제가 아니라 기능 기획을 실제 개발 가능한 작업 단위로 전환해야 하는 상태다.

가설:

- MVP 범위를 Agent + FRP TCP 터널 중심으로 좁히면 가장 빠르게 검증 가능한 기능 단위를 만들 수 있다.
- UDP, Nginx Stream, 직접 포트포워딩 자동화는 MVP 이후 확장 항목으로 분리하는 것이 안전하다.
- UDP와 Nginx Stream은 초기에는 자동 원격 수정 대상이 아니라 사용자 안내 가이드로 제공하는 것이 안전하다.
- 포트 할당, 인증, 상태 동기화, 매핑 회수 정책을 먼저 정리하지 않으면 구현 중 설계 변경 가능성이 크다.

## 영향 범위

현재 단계의 영향 범위는 문서화와 작업 계획에 한정한다.

향후 구현 시 예상 영향 범위:

- 설치형 프로그램 UI의 서버 공개 설정 화면
- Agent의 게임 서버 실행 및 포트 매핑 관리
- Relay/Jump 서버의 포트 할당 및 터널 관리
- Docker 게임 서버 포트 매핑
- 게임 템플릿의 프로토콜 및 포트 메타데이터
- 접속 주소 표시와 상태 동기화
- UDP, Nginx Stream 사용자 안내 가이드

## 검증 방법

현재 단계:

- Markdown 파일 UTF-8 strict 디코딩 확인
- 문서 구조 확인
- 필수 항목 포함 여부 확인

향후 구현 단계:

- Agent가 `frpc` 설정을 생성하는 단위 검증
- Relay 포트 할당 중복 방지 검증
- 게임 서버 시작 후 외부 접속 주소 생성 검증
- 게임 서버 종료 후 매핑 회수 검증
- 실제 TCP 접속 수동 검증은 현재 작업 공간에서는 수행하지 않고, 추후 Git 레포와 실행 환경이 추가된 뒤 검증 절차로 편입

## 단계별 작업 개요

1. 오늘 할 일 문서와 수행계획서를 작성한다.
2. 작업지시자의 수행계획서 승인을 받는다.
3. 승인 후 구현계획서를 작성한다.
4. 구현계획서에서 MVP 범위, 수정 예정 파일, 테스트 계획, 위험 요소를 확정한다.
5. 구현계획서 승인 후 2~5단계로 나누어 구현하고 단계별 보고서를 작성한다.
6. 모든 검증 후 최종 보고서를 작성한다.
