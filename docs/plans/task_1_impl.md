# task_1 구현계획서

## 분석 결과

현재 프로젝트는 GitHub 원격 저장소 `https://github.com/daewook0401/remote-game-server-app.git`에 연결되었다. 원격 저장소는 현재 기준으로 비어 있는 저장소처럼 보이며, 이 작업 공간의 문서를 시작점으로 프로젝트 구조를 잡는다.

이번 구현계획서는 즉시 애플리케이션 소스 코드를 수정하는 계획이 아니라, 실제 구현을 시작하기 전 개발 언어, 실행 환경, 파일 책임, 검증 기준을 확정하는 문서로 작성한다.

확정된 전제:

- 사용자 접점은 웹 UI가 아니라 설치형 프로그램 UI다.
- MVP는 Agent + FRP 기반 TCP 터널을 우선한다.
- 제품은 게임 서버 공개뿐 아니라 서버 연결, Minecraft 서버 생성, Docker 컨테이너 관리, 콘솔 접근까지 포함한다.
- 현재 작업 공간에서는 실제 TCP 접속 수동 검증을 수행하지 않는다.
- UDP, Nginx Stream, 직접 포트포워딩은 초기 자동화 범위가 아니라 사용자 안내 가이드 또는 후속 확장 범위다.
- 문서와 코드 파일은 UTF-8 인코딩을 유지해야 한다.

### 개발 언어 기준

이 프로젝트의 초기 개발 언어는 다음 기준으로 잡는다.

- 설치형 프로그램 UI: TypeScript
- 설치형 프로그램 UI 프레임워크: Electron + React
- 설치형 프로그램 내부 제어 계층: TypeScript
- Agent: Go
- Relay/Jump 서버 제어 API: Go
- Relay/Jump 서버 터널 처리: FRP(`frps`) 외부 바이너리 사용
- Agent 터널 처리: FRP(`frpc`) 외부 바이너리 사용
- 데이터 저장소 쿼리: SQL
- 컨테이너 실행 설정: Dockerfile, Docker Compose YAML
- 설정 파일: JSON 또는 TOML
- 문서와 사용자 안내 가이드: Markdown, 한국어

선택 이유:

- TypeScript는 설치형 프로그램 UI, 상태 관리, 화면 구성에 적합하다.
- Electron + React는 실제 데스크톱 프로그램 형태의 UI를 빠르게 구성하기 쉽다.
- Go는 Agent와 Relay처럼 프로세스 관리, 네트워크 상태 관리, 단일 바이너리 배포가 중요한 영역에 적합하다.
- FRP는 직접 구현하지 않고 검증된 외부 바이너리로 사용해 TCP 터널링 범위를 줄인다.

## 확정한 원인 또는 설계 방향

현재 문제는 네트워크 기능의 버그 수정이 아니라, 원격 게임 서버 공개 기능을 구현 가능한 MVP 단위로 좁히는 설계 작업이다.

설계 방향:

- 설치형 프로그램 UI에서 사용자가 로컬 서버, 기존 원격 서버, 클라우드 서버 중 대상 환경을 선택한다.
- 기존 원격 서버나 클라우드 서버는 SSH 접속 정보 또는 Agent 연결 정보를 등록한다.
- Agent는 대상 서버에서 Docker 상태를 확인하고 Minecraft 컨테이너 생성/시작/중지/삭제를 수행한다.
- 설치형 프로그램 UI는 Docker 로그와 콘솔 접근 기능을 제공한다.
- 설치형 프로그램 UI에서 사용자가 서버 공개 방식을 선택한다.
- MVP에서는 "점프 서버 방식"을 기본 경로로 제공한다.
- Agent는 게임 서버 실행 정보와 게임 템플릿의 포트 정보를 기준으로 `frpc` 설정을 생성한다.
- 중앙 Relay/Jump 서버는 공개 포트를 할당하고 `frps`를 통해 터널을 수신한다.
- 게임 서버 종료 또는 공개 중지 시 Agent와 중앙 서버가 포트 매핑을 정리한다.
- UDP와 Nginx Stream은 자동 수정 대신 사용자가 따라 할 수 있는 안내 문서를 프로그램 UI에 노출한다.

## 파일별 변경 계획

추후 실제 레포가 추가되면 아래 책임에 맞춰 파일을 배치한다.

### 설치형 프로그램 UI

- 대상 환경을 로컬 서버, 기존 원격 서버, 클라우드 서버로 구분한다.
- 원격 서버의 SSH 접속 정보 또는 Agent 연결 정보를 등록한다.
- Minecraft 서버 템플릿 선택과 생성 흐름을 제공한다.
- Docker 컨테이너 목록, 상태, 시작/중지/삭제 액션을 제공한다.
- Docker 로그와 콘솔 접속 화면을 제공한다.
- 네트워크 공개 방식 선택값을 저장한다.
- MVP에서는 Agent + FRP 방식을 기본 선택으로 노출한다.
- 직접 외부 오픈, UDP, Nginx Stream은 "가이드 보기" 중심으로 제공한다.
- 공개 상태를 `준비중`, `연결중`, `공개됨`, `실패`, `중지됨` 같은 명확한 상태로 표시한다.
- 공개 성공 시 `relay.example.com:port` 형식의 접속 주소를 보여준다.

### Agent

- 대상 서버에서 실행되며 설치형 프로그램과 백엔드 API의 명령을 수행한다.
- Docker 데몬 연결 상태를 확인한다.
- Minecraft 서버 컨테이너를 생성/시작/중지/삭제한다.
- 컨테이너 로그와 콘솔 입출력을 중계한다.
- 게임 인스턴스의 내부 포트와 프로토콜을 읽는다.
- 중앙 서버에 공개 포트 할당을 요청한다.
- 할당받은 포트를 기준으로 `frpc` 설정을 생성한다.
- `frpc` 실행 상태를 감시한다.
- 게임 서버 종료 시 `frpc`를 중지하고 중앙 서버에 매핑 회수를 요청한다.

### Relay/Jump 서버

- 공개 포트 풀을 관리한다.
- 동일 포트 중복 할당을 방지한다.
- Agent 인증 토큰을 검증한다.
- 포트 매핑 테이블에 사용자, 서버, 게임 인스턴스, 내부 포트, 외부 포트, 상태를 저장한다.
- 매핑 회수 실패 시 재시도 또는 정리 작업 대상에 넣는다.

### 게임 템플릿

- 게임별 기본 내부 포트와 프로토콜을 정의한다.
- MVP에서는 Minecraft Java TCP `25565`를 우선 대상으로 삼는다.
- UDP가 필요한 게임은 자동 공개 전에 안내 또는 미지원 상태를 표시한다.

### 안내 문서

- 원격 서버 접속 정보 등록과 Agent 설치 절차를 설명한다.
- Amazon 같은 클라우드 서버에서 방화벽/보안 그룹/SSH 접속을 준비하는 절차를 설명한다.
- UDP는 게임별 요구 포트와 공유기/방화벽 설정 필요성을 설명한다.
- Nginx Stream은 외부망 서버와 내부망 서버가 있는 사용자를 대상으로 예시 설정을 제공한다.
- 직접 포트포워딩은 공유기 포트포워딩, 방화벽/UFW, Docker 포트 매핑 확인 절차를 제공한다.

- Agent의 `frpc` 설정 생성 테스트

- 설치형 프로그램 UI 상태 전이 테스트

- TCP 접속 수동 검증

현재 환경에서 제외하는 검증:

- 실제 외부 클라이언트에서 Relay 포트로 접속하는 수동 TCP 검증
- 실제 UDP 게임 서버 접속 검증
- 실제 Nginx Stream 원격 적용 검증

## 위험 요소

- Relay 서버를 통한 트래픽은 대역폭 비용과 운영 비용이 발생한다.
- 포트 매핑 회수 실패 시 유령 매핑이나 포트 고갈이 발생할 수 있다.
- Agent 인증이 약하면 임의 포트 공개나 터널 악용 위험이 있다.
- `frpc` 설정 파일에 민감 정보가 들어갈 수 있으므로 저장 권한과 삭제 정책이 필요하다.
- UDP 지원은 게임별 차이가 커서 MVP에 포함하면 범위가 커질 수 있다.
- Nginx Stream은 사용자 인프라에 직접 영향을 주므로 자동 수정보다 안내 중심 접근이 안전하다.
- 현재 작업 공간에서는 실제 TCP 접속을 검증할 수 없어, 추후 레포와 실행 환경에서 검증 절차를 반드시 보강해야 한다.

## 승인 요청

이 구현계획서 기준으로 1단계 문서 작업을 완료 처리하고, 추후 실제 Git 레포가 추가되면 2단계 구조 분석부터 진행해도 되는지 승인을 요청한다.
