# task_2 수행계획서

## 작업 제목

서버 선택 중심 내비게이션 및 게임 서버별 콘솔 진입 구조 정리

## 목표

- 사이드바 메뉴에서 `콘솔`, `외부 공개`를 제거하고 `서버 선택`, `안내 가이드`만 남긴다.
- 서버 선택 화면을 최상위 부모 흐름으로 보고, 서버 상세 화면 안에서 게임 서버 목록과 콘솔 접근을 처리한다.
- 서버 상세 화면에 들어가면 `Agent 상태 확인`을 자동으로 한 번 실행해 서버 상태와 Docker 컨테이너 목록을 자연스럽게 불러온다.
- Docker 컨테이너 목록의 각 게임 서버 옆에 `콘솔` 버튼을 추가해 해당 컨테이너 콘솔로 바로 이동하거나 표시할 수 있게 한다.
- 서버 카드의 `상태 확인` 버튼 제거를 검토하고, 필요한 상태 확인은 서버 상세 진입 시 자동 확인 및 기존 준비 영역에서 처리한다.
- Agent 18080 포트는 Docker 제어 권한을 가진 관리 API이므로, 외부 공개 상태와 인증 토큰 누락 위험을 UI/계획에서 명확히 다룬다.
- 원격/클라우드 서버 Agent token은 사용자가 직접 다루지 않아도 앱이 자동 생성하고 설치 스크립트와 로컬 저장 정보에 함께 적용하는 방향으로 검토한다.
- 서버 삭제 시 원격 Agent 서비스, Agent 데이터, `18080/tcp` 방화벽 허용 규칙 삭제 옵션을 제공하고, 원격 정리 후 로컬 등록 정보를 삭제하는 순서를 검토한다.

## 현재 현상 또는 요구사항

- 현재 사이드바는 `서버 선택`, `콘솔`, `외부 공개`, `안내 가이드` 4개 메뉴로 구성되어 있다.
- `콘솔`은 독립 페이지로 열리고, 그 안에서 서버와 컨테이너를 다시 선택해야 한다.
- `외부 공개`는 독립 메뉴로 노출되어 있으나 이번 방향에서는 필요하지 않다.
- 서버 카드에는 `들어가기`, `상태 확인`, `삭제` 버튼이 함께 노출된다.
- 사용자는 서버가 없으면 콘솔/외부 공개 기능도 작동 흐름상 의미가 약하므로, 서버 선택을 부모 트리로 두는 구조를 원한다.
- 사용자는 서버 선택 후 상세 화면에 들어갔을 때 Agent 상태 확인이 자동으로 한 번 실행되는 UX를 원한다.
- 사용자는 테스트 서버에 설치한 18080 Agent 포트가 보안 문제가 될 수 있다고 우려한다.
- 사용자는 Agent token 발급/적용을 프로그램이 보이지 않는 곳에서 자동 처리하길 원한다.
- 사용자는 서버 삭제 시 원격 Agent와 관련 데이터도 함께 삭제되길 원한다.

## 재현 방법 또는 확인 방법

1. 데스크톱 앱을 실행한다.
2. 좌측 사이드바에 `콘솔`, `외부 공개` 메뉴가 보이는지 확인한다.
3. `서버 선택` 화면에서 등록된 서버 카드의 `상태 확인` 버튼이 보이는지 확인한다.
4. 서버 상세로 들어간 뒤 Agent 상태 확인을 사용자가 직접 눌러야 상태와 컨테이너 목록이 갱신되는지 확인한다.
5. 서버 상세로 들어간 뒤 Docker 컨테이너 목록의 각 게임 서버 행에 콘솔 진입 버튼이 없는지 확인한다.

## 원인 가설

- 초기 UI가 기능 단위 페이지(`콘솔`, `외부 공개`) 중심으로 구성되어 서버 선택 흐름과 분리되어 있다.
- 콘솔 페이지가 저장된 서버 목록을 독립적으로 다시 불러오는 구조라, 서버 상세 화면의 컨테이너 행과 직접 연결되어 있지 않다.
- 외부 공개 기능이 메뉴와 페이지로 별도 노출되어 현재 목표보다 넓은 기능 범위를 갖고 있다.
- 서버 상세 진입 시 Agent 상태 확인이 자동 실행되지 않아, 사용자가 서버를 선택한 뒤에도 다시 상태 확인 버튼을 눌러야 컨테이너 목록을 확인할 수 있다.
- 현재 원격 Agent 자동 설치 스크립트는 `AGENT_ADDR=0.0.0.0:18080` 구성을 사용하므로, 방화벽까지 열려 있으면 Agent API가 외부 네트워크에 노출될 수 있다.
- `AGENT_TOKEN`이 비어 있으면 `/docker/*` API가 인증 없이 동작한다.
- 현재 서버 삭제는 앱의 등록 서버 목록에서만 제거하며, 원격 서버의 Agent 서비스와 `/opt/remote-game-agent` 데이터는 정리하지 않는다.
- 현재 서버 삭제는 Agent 관리 포트 `18080/tcp` 방화벽 허용 규칙도 정리하지 않는다.

## 검증 방법

- `npm run desktop:typecheck`
- `npm run desktop:build`
- 필요 시 데스크톱 개발 서버 또는 Electron 실행으로 화면 수동 확인
- 확인 포인트:
  - 사이드바에 `서버 선택`, `안내 가이드`만 보인다.
  - 서버 카드에서 `상태 확인` 버튼이 사라진다.
  - 서버 상세 화면에 들어가면 Agent 상태 확인이 자동으로 한 번 실행되고, 서버 상태와 컨테이너 목록 갱신을 시도한다.
  - 자동 확인 중복 실행으로 사용자가 의도하지 않은 반복 요청이 발생하지 않는다.
  - 서버 상세의 Docker 컨테이너 행에 `콘솔` 버튼이 보인다.
  - 콘솔 버튼은 해당 서버와 컨테이너를 대상으로 콘솔 로그 조회 및 명령 전송 흐름에 연결된다.
  - 외부 공개 메뉴로 접근할 수 없다.
  - Agent URL이 외부 IP 또는 토큰 없는 구성일 때 보안상 위험을 인지할 수 있다.
  - 원격/클라우드 Agent 설치 시 토큰이 자동 생성되어 서버와 로컬 저장 정보에 적용된다.
  - 서버 삭제 시 원격 Agent 정리 옵션을 선택할 수 있고, Agent 서비스/데이터와 `18080/tcp` 방화벽 규칙 정리 후 로컬 등록 정보가 삭제된다.

## 단계별 작업 개요

1. 구현계획 수립
   - 독립 콘솔 페이지를 제거할지, 콘솔 UI를 재사용 컴포넌트로 분리할지 확정한다.
   - 외부 공개 관련 코드를 메뉴에서만 숨길지, 라우트와 import까지 정리할지 확정한다.
   - Agent 18080 포트 보안 조치를 이번 작업에 포함할 범위와 후속 작업으로 분리할 범위를 확정한다.
   - Agent token 자동 생성/적용 범위와 서버 삭제 시 Agent 정리 범위를 확정한다.
2. 내비게이션 정리
   - 사이드바 메뉴와 라우트 타입에서 `console`, `publish` 노출을 제거한다.
   - `App.tsx`의 제거 라우트 렌더링을 정리한다.
3. 서버 선택 카드 정리
   - `ServerCard`에서 `상태 확인` 버튼과 관련 props를 제거한다.
   - 서버 상세 진입 시 `Agent 상태 확인`을 자동으로 한 번 실행하도록 흐름을 설계한다.
   - 사용자가 필요하면 서버 상세 안의 `Agent 상태 확인` 버튼으로 수동 재확인할 수 있게 유지한다.
4. 서버 상세 자동 상태 확인 추가
   - 선택된 서버 상세 진입 또는 서버 전환 시 최초 1회 Agent 상태와 컨테이너 목록을 자동 조회한다.
   - 같은 서버에서 렌더링이 반복되어도 중복 요청이 발생하지 않도록 기준을 둔다.
5. 게임 서버별 콘솔 진입 추가
   - `ContainerTable` 작업 영역에 `콘솔` 버튼을 추가한다.
   - `ServerManagementPage`에서 선택 서버와 선택 컨테이너를 기준으로 콘솔 패널 또는 모달을 띄운다.
6. 검증 및 문서화
   - 타입체크/빌드를 실행한다.
   - 단계별 보고서와 최종 보고서를 작성한다.
   - 오늘 할 일 상태를 갱신한다.

## 승인 요청

위 계획으로 진행해도 되는지 승인 요청드립니다.

특히 구현계획서에서 아래 방향을 확정하려고 합니다.

- 콘솔은 서버 상세 화면 안에서 모달 또는 인라인 패널로 제공한다.
- `외부 공개`는 사이드바/라우트에서 제거하되, 내부 서비스 코드는 이번 범위에서 사용 여부를 확인한 뒤 최소 정리한다.
- 서버 카드의 `상태 확인` 버튼은 제거하고, 서버 상세 진입 시 Agent 상태 확인을 자동으로 한 번 실행한다.
- 상태 확인 수동 재실행은 서버 상세의 `Agent 상태 확인` 버튼으로 유지한다.
- Agent 18080 포트는 기본적으로 외부 공개하지 않는 방향을 우선하며, 불가피하게 공개할 경우 토큰과 방화벽 제한을 요구한다.
- Agent token은 앱이 자동 생성/적용하고, UI에는 원문 대신 설정 상태 중심으로 노출한다.
- 서버 삭제 시 순서는 `원격 Agent 서비스 종료/삭제`, `18080/tcp 방화벽 규칙 삭제`, `Agent 데이터 삭제`, `로컬 SSH/Agent 등록 정보 삭제`를 기본으로 검토한다.
