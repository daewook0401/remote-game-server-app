# task_2 최종 보고서

## 배경

서버 선택을 부모 흐름으로 두고, 독립 메뉴였던 콘솔과 외부 공개를 제거하는 방향으로 UI를 정리했다. 이후 Agent 관리 포트 `18080`의 보안 위험이 확인되어 토큰 자동 생성/적용, 위험 구성 경고, 서버 삭제 시 Agent 정리 흐름을 함께 반영했다.

## 원인 또는 설계 판단

- 콘솔과 외부 공개가 독립 메뉴로 노출되어 서버 선택 흐름과 분리되어 있었다.
- Agent API는 Docker 제어 권한을 가진 관리 API이므로 일반 공개 포트처럼 다루면 안 된다.
- 원격 Agent 접근은 장기적으로 SSH 터널이 권장되지만, 이번 작업에서는 토큰 없는 공개 구성을 줄이는 조치를 우선 반영했다.
- 서버 삭제 시 로컬 등록 정보만 삭제하면 원격 Agent와 `18080` 포트 정책이 남을 수 있어 삭제 흐름에 원격 정리를 추가했다.

## 변경 내용

- 사이드바 메뉴를 `서버 선택`, `안내 가이드`로 축소했다.
- `console`, `publish` 라우트 렌더링을 제거했다.
- 서버 카드의 `상태 확인` 버튼을 제거했다.
- 서버 상세 진입 시 Agent 상태와 Docker 컨테이너 목록을 자동으로 한 번 조회한다.
- Docker 컨테이너 행에 `콘솔` 버튼을 추가하고, 서버 상세 모달에서 컨테이너 콘솔을 볼 수 있게 했다.
- 외부 Agent URL인데 token이 비어 있으면 보안 경고를 표시한다.
- 원격/클라우드 Agent 설치 시 token을 자동 생성해 설치 요청과 로컬 저장 정보에 적용한다.
- 서버 삭제 시 원격 Agent 서비스, `18080/tcp` 서버 내부 방화벽 규칙, Agent 데이터 정리 후 로컬 등록 정보를 삭제하는 옵션을 추가했다.

## 영향 범위

- 데스크톱 앱 라우팅과 사이드바 메뉴
- 서버 선택/상세 화면
- Docker 컨테이너 목록 작업 버튼
- Agent 설치/삭제 Electron IPC
- 서버 등록 UI의 Agent 인증 표시

## 남은 위험

- 실제 원격 서버에서 Agent 설치 token 적용, 콘솔 명령, Agent 삭제, 18080 방화벽 규칙 삭제는 사용자 수동 테스트가 필요하다.
- 클라우드 보안 그룹과 공유기 포트포워딩의 18080 규칙은 앱이 자동 삭제하지 못할 수 있다.
- 자동 생성 token은 현재 로컬 저장소에 저장되므로 OS 보안 저장소 연동은 후속 작업이 필요하다.
- SSH 터널 기반 Agent 접속 구조는 별도 task로 설계해야 한다.

## 교훈

- Agent 관리 포트와 게임 서버 공개 포트는 UI와 문서에서 명확히 분리해야 한다.
- 서버 등록 정보 삭제 전에 원격 리소스 정리 여부를 먼저 결정해야 한다.
- 보안 기본값은 사용자가 직접 토큰을 다루지 않아도 안전한 방향이어야 한다.

## 후속 작업

- 사용자 수동 테스트 피드백 반영
- SSH 터널 기반 Agent 접속 구조 계획 및 구현
- Agent token OS 보안 저장소 연동 검토
- 직접 공개 예외 모드의 방화벽 허용 IP 제한 UX 설계
