# Task 1 피드백: 비전문가용 게임 서버 생성 흐름

## 기록일

2026-04-28

## 피드백 요약

- 첫 화면과 주요 흐름은 개발자용 Agent/Docker 관리가 아니라 게임 선택에서 시작해야 한다.
- 사용자는 게임을 고른 뒤 서버를 어디에 만들지 선택한다.
  - 현재 실행 중인 PC
  - 등록된 SSH 서버
  - 클라우드 서버
- SSH 서버를 등록하면 Agent URL은 사용자가 직접 입력하지 않고 SSH host와 고정 포트 `18080`으로 자동 구성한다.
- Docker 내부 포트는 게임 권장 포트로 고정하고 사용자가 수정하지 못하게 한다.
  - Minecraft Java 기본 내부 포트는 `25565`.
  - 사용자는 외부 공개 포트만 수정한다.
- 서버 생성 버튼을 누르면 원격 서버의 방화벽 포트를 열 수 있음을 명확히 경고해야 한다.
  - 포트 개방에는 sudo 권한이 필요하다.
  - SSH 비밀번호를 입력받아 sudo 입력으로 재사용한다.
  - 비밀번호는 저장하지 않는다.
- 서버 볼륨은 서비스 전용 경로 아래에 만들어야 한다.
  - 기본 설계: `/remote-game-server/volume/{game}/{serverName}`.
  - 추후 실제 사업 아이템 이름이 정해지면 루트 경로를 교체한다.
- 컨테이너 삭제는 단일 삭제 버튼으로 끝내지 않는다.
  - 컨테이너만 삭제: 맵 데이터, 설정, 로그를 보존한다.
  - 전체 데이터 삭제: 컨테이너와 맵 데이터, 설정, 로그를 모두 삭제한다.
  - 전체 삭제는 강한 경고와 확인 입력이 필요하다.
- 알림은 시스템 팝업이 아니라 앱 내부 UI로 처리한다.
  - 일반 알림은 자동으로 사라지되 X 버튼으로 즉시 닫을 수 있어야 한다.
  - 비밀번호 입력, 경고, 확인은 앱 내부 모달로 처리한다.
- 전체 UI 문구와 흐름은 전문 지식이 적은 사용자도 따라갈 수 있게 해야 한다.

## 결정 사항

- 원격 Agent 기본 주소는 `http://{sshHost}:18080`으로 자동 생성한다.
- Agent 설치와 Agent 업데이트는 계속 분리한다.
- Agent 설치/업데이트 시 기존 게임 서버 정보는 Agent의 상태 파일에 보존한다.
- 이 단계의 변경은 우선 로컬 테스트용으로 반영하고, Git commit/push 및 Release 갱신은 별도 승인 후 진행한다.
